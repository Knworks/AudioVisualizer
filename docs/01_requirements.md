# 🧾AudioVisualizer 要件定義書

## 1. 🎯背景と目的

* WPF アプリケーションへ組み込み可能な、再利用性の高い音声可視化コンポーネントを提供する。
* システム再生音およびマイク入力を視覚的に表現し、利用側アプリケーションで直感的に扱えるようにする。
* 公開プロパティ設定や ViewModel バインドで利用できる構成とし、利用側の実装負荷を下げる。
* 可視化エフェクトの切替と拡張を可能にし、将来的な組込エフェクト追加や外部エフェクトパッケージ配布に発展できる構造を整える。
* 初期提供では代表的な組込エフェクト 5 種類を提供し、用途ごとに見え方を切り替えられる状態にする。

## 2. 👥利用者とステークホルダー

* 利用者

  * WPF アプリケーションに音声可視化機能を組み込みたい開発者
  * MVVM パターンで UI を構築する開発者
  * 独自エフェクトや独自デバイス選択 UI を実装したい開発者
* ステークホルダー

  * コンポーネント開発者
  * SampleApp を通じて動作確認や利用方法確認を行う OSS 利用者
  * GitHub 上でソースコード、サンプル、ドキュメントを参照する利用者

## 3. 📚用語・ドメイン定義

| 用語               | 定義                                         | 備考                              |
| ---------------- | ------------------------------------------ | ------------------------------- |
| AudioVisualizer  | 本要件で定義するソリューション全体の名称                       | GitHub 公開対象                     |
| コンポーネント          | `AudioVisualizer.Wpf` が提供する WPF 用音声可視化部品   | `CustomControl` として提供           |
| CustomControl    | WPF の `Control` 派生コントロール                   | `UserControl` は採用しない            |
| Core             | 音声取得の抽象、分析処理、モデルなどの WPF 非依存ロジック層           | `AudioVisualizer.Core`          |
| Wpf              | WPF 向けの描画、コントロール、DependencyProperty を担う層   | `AudioVisualizer.Wpf`           |
| SampleApp        | コンポーネントの利用例と動作確認を行うサンプルアプリ                 | `AudioVisualizer.SampleApp`     |
| InputSource      | 音声入力種別                                     | `SystemOutput` または `Microphone` |
| SystemOutput     | Windows の再生音を対象とする入力種別                     | 再生デバイスのループバック入力を想定              |
| Microphone       | Windows のマイク入力を対象とする入力種別                   | 入力デバイスを対象                       |
| DeviceId         | 利用する音声デバイスを識別する一意 ID                       | 表示名ではなく識別子を用いる                  |
| UseDefaultDevice | OS の既定デバイスを利用する設定                          | `true` の場合は既定デバイス変更へ追従          |
| IsActive         | 音声取得および可視化処理の開始/停止状態                       | `true` の場合に処理を開始                |
| エフェクト            | 音声データを可視化表現へ変換する描画ロジック                     | 例: バー型スペクトラム、波形                 |
| 組込エフェクト          | ライブラリ標準で提供するエフェクト                          | `SpectrumBar`、`WaveformLine`、`MirrorBar`、`PeakHoldBar`、`BandLevelMeter` |
| 独自エフェクト          | 利用側が独自実装して追加するエフェクト                        | オブジェクト差し替えで拡張                   |
| 外部エフェクトパッケージ     | ライブラリ外部で配布され、後から追加可能なエフェクト群またはエフェクト実装パッケージ | MVP では読込機能自体は対象外                |
| エフェクトメタデータ       | エフェクトの識別、表示、互換性判定に必要な情報                    | 例: ID、表示名、作者、バージョン、説明、対応入力種別    |
| エフェクトプロバイダ       | エフェクト本体とそのメタデータを提供する単位                     | 将来の外部配布単位として利用可能                |
| ホストバージョン         | コンポーネント本体の互換性判定に用いるバージョン                   | 外部エフェクトとの互換性確認に利用               |
| VisualizerFrame  | 1 フレーム分の可視化用分析結果                           | スペクトラム、波形、ピーク値などを含む             |
| デバイス選択 UI        | 音声デバイス一覧を表示し、利用デバイスを選択する UI                | MVP ではコンポーネントに含めない              |

## 4. 🎭ユースケース / ユーザーストーリー

* 開発者として、WPF アプリケーションの XAML にコンポーネントを配置して、少ない設定で音声可視化を利用したい。
* 開発者として、ViewModel から `InputSource`、`Effect`、`IsActive` などをバインドし、画面状態に応じて可視化を制御したい。
* 開発者として、同じコンポーネントでシステム再生音とマイク入力の両方を扱いたい。
* 開発者として、用途に応じて可視化エフェクトを簡単に切り替えたい。
* 開発者として、波形、対称バー、ピーク保持、帯域メーターを用途に応じて選択したい。
* 開発者として、ライブラリ本体を改修せずに独自エフェクトを追加したい。
* 開発者として、将来的に外部エフェクトパッケージを導入できる構造を前提に開発したい。
* 開発者として、設定画面やオプション画面で使うために、再生デバイス一覧やマイクデバイス一覧を API から取得したい。
* 開発者として、デバイス選択 UI は自分で実装し、コンポーネントにはデバイス取得・設定 API だけを持たせたい。
* 開発者として、表示領域のサイズに応じて自動でスケールし、どこに配置しても違和感の少ない見た目にしたい。
* 開発者として、低域偏重を避けたい場面に備えて、スペクトラムのバー割り当て方式を設定で切り替えたい。
* OSS 利用者として、GitHub 上の SampleApp とソースコードから利用方法を理解したい。
* 開発者として、TDD を前提に Core と Wpf を分離し、テストしやすい構造で実装したい。

## 5. ✅機能要件

* WPF ベースの再利用可能なコンポーネントとして提供すること。
* コンポーネントは `UserControl` ではなく `CustomControl` を主軸として実装すること。
* 利用側アプリケーションから XAML 配置できること。
* 公開プロパティを通じて設定可能であること。
* ViewModel とのデータバインドで利用可能であること。
* 音声入力元として以下をサポートすること。

  * システム再生音
  * マイク入力
* 音声取得と可視化ロジックを分離すること。
* バー型スペクトラムの計算プロファイルを公開プロパティで切り替え可能であること。
* 可視化エフェクトは列挙型ではなく、オブジェクト差し替え方式で切り替え可能であること。
* 初期提供として、以下の組込エフェクトを提供すること。

  * `SpectrumBarEffect`
  * `WaveformLineEffect`
  * `MirrorBarEffect`
  * `PeakHoldBarEffect`
  * `BandLevelMeterEffect`
* 将来的に、再生音向け 3〜5 種類、マイク入力向け 3〜5 種類の組込エフェクトを追加可能な構造であること。
* 利用側が独自エフェクトを追加できる拡張ポイントを提供すること。
* 将来的な外部エフェクトパッケージ配布に対応できるよう、エフェクト拡張点を明確に分離した構造を持つこと。
* エフェクトは、描画ロジック本体と識別・説明用のメタデータを扱える構造であること。
* エフェクトは、少なくとも以下のメタデータを保持可能であること。

  * 一意な識別子
  * 表示名
  * バージョン
  * 作者情報
  * 説明
  * 対応する入力種別
* 組込エフェクトと将来の外部エフェクトを、利用側から同一の概念で扱えること。
* 将来的に、外部エフェクトパッケージの一覧取得、有効化、無効化に拡張可能な構造を持つこと。
* 将来的に、ホスト側バージョンと外部エフェクトの互換性を判定できるようにすること。
* 外部エフェクトの読込に失敗した場合でも、本体コンポーネント全体が異常終了しにくい構造とすること。
* MVP では外部エフェクトパッケージの読込機能自体は実装対象外とするが、将来追加時に既存の組込エフェクト契約を大きく変更しなくてよい構造とすること。
* 利用可能な音声デバイス一覧を API として取得できること。
* デバイス一覧 API には以下を含むこと。

  * `SystemOutput` 用デバイス一覧取得
  * `Microphone` 用デバイス一覧取得
  * 各入力種別の既定デバイス取得
* 利用側から `DeviceId` を指定して使用デバイスを切り替えられること。
* `UseDefaultDevice = true` のとき、OS の既定デバイス変更に追従すること。
* `UseDefaultDevice = false` のとき、明示指定した `DeviceId` を使用すること。
* `InputSource`、`DeviceId`、`UseDefaultDevice` の変更時は、`IsActive = true` のときのみ再接続すること。
* `IsActive = false` のときは再接続せず、設定値のみ保持すること。
* コントロールは配置領域のサイズに応じて自動調整されること。
* 小さい領域でも表示が破綻しにくく、大きい領域でも違和感が少ないこと。
* ソリューションは以下のプロジェクト構成を前提とすること。

  * `AudioVisualizer.Core`
  * `AudioVisualizer.Wpf`
  * `AudioVisualizer.SampleApp`
  * `AudioVisualizer.Core.Tests`
  * `AudioVisualizer.Wpf.Tests`

## 6. 🛡️非機能要件

### 🎨デザイン

* 表示領域のサイズに応じて自動調整され、どこに配置しても違和感の少ない見た目とする。
* 固定サイズ前提ではなく、配置先サイズに応じてレスポンシブに描画できること。
* 利用側アプリケーションの UI に馴染みやすい、汎用的な表示を実現できること。

### 🖱️ユーザビリティ

* 利用側開発者が XAML 配置、公開プロパティ設定、ViewModel バインドで直感的に扱えること。
* デバイス選択 UI はコンポーネントに内包せず、利用側が必要に応じて自由に実装できる API 設計であること。
* 入力元切替、エフェクト切替、開始/停止の制御が分かりやすいこと。
* SampleApp から組込エフェクトを切り替え、見え方の違いを確認できること。
* 組込エフェクトと将来の外部エフェクトを同じ考え方で扱えること。

### 🏷️ブランド要件

* GitHub 上に公開しやすい名称、構成、サンプルを持つこと。
* ソリューション名は `AudioVisualizer` とすること。
* プロジェクト名は以下を採用すること。

  * `AudioVisualizer.Core`
  * `AudioVisualizer.Wpf`
  * `AudioVisualizer.SampleApp`
  * `AudioVisualizer.Core.Tests`
  * `AudioVisualizer.Wpf.Tests`

### ⚡パフォーマンス最適化

* 通常利用時に過度な CPU 使用率とならないこと。
* 音声取得と描画処理を分離し、UI のカクつきを抑えること。
* 連続的な可視化更新に耐えられること。
* `IsActive = false` 時には不要な音声取得や再接続処理を行わないこと。
* 将来的に外部エフェクトを追加した場合でも、エフェクト読込や切替によって全体の応答性が著しく低下しにくい構造であること。

### 🔐セキュリティ

* 現時点で明示要件なし。
* 将来的な外部エフェクトパッケージ読込を見据え、信頼性確認や安全な読込方式を追加可能な構造であること。

### 🟢可用性

* 現時点で明示要件なし。
* 将来的に外部エフェクトの読込失敗や互換性不整合が発生しても、本体全体が異常終了しにくい構造であること。

### 🛠️保守性

* `Core` と `Wpf` を分離し、WPF 依存を UI 層へ閉じ込めること。
* TDD を前提に、テストしやすい責務分離を行うこと。
* `AudioVisualizer.Core.Tests` で Core の仕様を検証できること。
* `AudioVisualizer.Wpf.Tests` で WPF 固有の振る舞いを検証できること。
* エフェクト追加や将来の外部エフェクトパッケージ拡張に対応しやすい構造であること。
* エフェクトの契約、メタデータ、描画責務を分離し、拡張時の変更範囲を局所化できること。
* 組込エフェクトと外部エフェクトを同一の拡張モデルで扱えること。
* SampleApp を通じて利用方法を確認できること。

## 7. 🔒制約条件

### 開発環境や言語、フレームワークに制約

* Windows 向け WPF コンポーネントとして開発すること。
* 実装言語は C# を前提とすること。
* UI フレームワークは WPF を採用すること。
* `UserControl` ではなく `CustomControl` を主軸とすること。
* 音声取得と可視化は分離設計とすること。
* エフェクトは列挙型ではなくオブジェクト差し替えを基本とすること。
* TDD を前提に、テストプロジェクトを含む構成で開発すること。

### 外部システムとの連携

* 外部クラウドサービスや業務システムとの連携は不要とする。
* MVP では外部エフェクトパッケージの読込機能は不要とする。
* 連携対象は Windows の音声デバイスおよび OS の既定デバイス状態とする。
* GitHub で公開することを前提とする。

## 8. ⚠️リスクと課題

### 想定されるリスクや懸念点

* Windows の音声デバイス状態は実行環境に依存するため、利用環境ごとの差異が発生する可能性がある。
* OS の既定デバイス変更追従は、環境差やデバイス切断時の挙動確認が必要である。
* 可視化エフェクトの自由度を高くすると、描画責務と拡張ポイントの境界が曖昧になりやすい。
* サイズ自動調整に対応するため、極小サイズや極端な縦横比での見え方に配慮が必要である。
* `Core` に WPF 依存型が混入すると、責務分離と TDD の前提が崩れるリスクがある。
* 将来的に外部エフェクトパッケージを許可する場合、互換性不整合、読込失敗、性能劣化、例外発生などのリスクがある。
* エフェクト契約が単純すぎると、後からメタデータや互換性情報を追加する際に破壊的変更が必要になるおそれがある。
* GitHub 公開を前提とするため、プロジェクト構成、サンプル、ドキュメントが不十分だと利用者に伝わりにくい。

### 解決すべき前提条件や依存関係

* `AudioVisualizer.Core` は WPF 非依存であることを前提とする。
* `AudioVisualizer.Wpf` は `AudioVisualizer.Core` に依存し、描画責務を担うことを前提とする。
* デバイス一覧取得や既定デバイス通知は API として提供し、UI 実装は利用側責務とする。
* エフェクトは将来の外部配布を見据え、描画契約とメタデータ契約を分離できる前提で設計する。
* 組込エフェクトと外部エフェクトを同一の取得・管理モデルへ統合できるようにする。
* 外部エフェクトパッケージの読込機能は MVP 対象外とし、まずは契約と拡張点の安定化を優先する。
* 初期提供では 5 種類の組込エフェクトを同一契約で実装し、切替時も既存の入力・描画責務が破綻しないことを優先する。
* SampleApp は、コンポーネント利用方法およびデバイス選択 UI 実装例を示す前提で用意する。
